1. Contexto del ejercicio
Una fintech esta construyendo un nuevo sistema de originacion de creditos. Tu tarea es desarrollar el
microservicio que recibe solicitudes de credito de los clientes, calcula la cuota mensual estimada con base
en una tasa anual fija, y enriquece la solicitud con el equivalente del monto solicitado en otras divisas
(USD y EUR) consultando una API publica de tipos de cambio.

La solicitud se persiste en PostgreSQL y puede cambiar de estado a lo largo del flujo: CREATED (recien
capturada) -> UNDER_REVIEW (analista la esta revisando) -> APPROVED / REJECTED / CANCELLED.
La empresa ha definido como estandar la arquitectura hexagonal (ports and adapters) para garantizar que
la logica del dominio sea independiente de frameworks y de la base de datos, y para poder testear sin
depender de la API externa. Todo el entorno debe ser levantable localmente con un solo comando
(docker-compose up).

3. API publica a integrar: Frankfurter
Vas a consumir la API publica Frankfurter para obtener tipos de cambio. Es gratuita, no requiere API key
y entrega datos del Banco Central Europeo (ECB).
Campo Valor
Sitio oficial https://frankfurter.dev
Base URL https://api.frankfurter.dev/v1
Autenticacion Ninguna (es publica y gratuita)
Rate limit Razonable para pruebas - cachea cuando puedas
3.1 Endpoint a usar
Obtener el ultimo tipo de cambio MXN a USD y EUR:
GET https://api.frankfurter.dev/v1/latest?base=MXN&symbols=USD,EUR
# Respuesta de ejemplo:
{
 "amount": 1.0,
 "base": "MXN",
 "date": "2026-05-11",
 "rates": {
 "EUR": 0.052,
 "USD": 0.058
 }
}
3.2 Como debes usarla en este ejercicio
• Cada vez que se crea una nueva solicitud de credito, llama a Frankfurter para obtener la tasa
MXN -> USD y MXN -> EUR del dia.
• Calcula el equivalente del monto solicitado en USD y EUR y guardalo en la base de datos.
• Si la API esta caida o tarda mas de 3 segundos, NO debe romper la creacion. Guarda los campos
como null y registra un warning en logs (los campos son informativos, no criticos).
• Implementa la llamada detras de un puerto del dominio: el dominio NO debe saber que existe
Frankfurter.
• Idealmente cachea la respuesta por 1 hora para no llamar cada vez (bonus).
Implementacion del puerto
Define en el dominio una interfaz ExchangeRateProvider con un metodo 'getRates(baseCurrency,
targetCurrencies)'. Crea un adapter FrankfurterExchangeRateAdapter en infrastructure que la
implemente usando RestClient o WebClient. Asi puedes mockearlo facil en pruebas y, si manana
cambia el proveedor, solo se reemplaza el adapter.

4. Especificaciones funcionales (la API)
La API expone los siguientes endpoints. Todos en formato JSON.
4.1 POST /api/v1/credit-applications - Crear solicitud
Crea una nueva solicitud en estado CREATED.
Request body:
{
 "customerName": "Juan Perez Ramirez",
 "customerEmail": "juan.perez@example.com",
 "customerRfc": "PERJ850101ABC",
 "requestedAmount": 250000.00,
 "currency": "MXN",
 "termMonths": 24,
 "annualInterestRate": 0.18
}
Reglas de validacion:
• customerName: obligatorio, 3 a 120 caracteres.
• customerEmail: obligatorio, formato email.
• customerRfc: opcional, si viene debe tener 12 o 13 caracteres alfanumericos.
• requestedAmount: obligatorio, > 0, maximo 5,000,000.
• currency: obligatorio, default 'MXN'. Solo se acepta 'MXN' por ahora.
• termMonths: obligatorio, entero entre 6 y 60.
• annualInterestRate: obligatorio, decimal entre 0.05 y 0.60 (es decir 5% a 60% anual).
Logica del dominio al crear:
1. Calcular la cuota mensual con formula de amortizacion francesa:
r = annualInterestRate / 12
monthlyPayment = P * r / (1 - (1 + r)^-n)
 donde P = requestedAmount, n = termMonths
totalToPay = monthlyPayment * termMonths
2. Consultar Frankfurter para obtener tasa MXN -> USD y MXN -> EUR.
3. Calcular y guardar amountUsd = requestedAmount * rateUsd.
4. Calcular y guardar amountEur = requestedAmount * rateEur.
5. Persistir la solicitud con status = CREATED.
6. Responder 201 Created con el body completo de la solicitud creada.
4.2 GET /api/v1/credit-applications/{id} - Consultar una solicitud
• Si existe: 200 con el body completo.
• Si no existe: 404 con cuerpo de error consistente.
4.3 GET /api/v1/credit-applications - Listar solicitudes
• Filtros opcionales (query params): status, customerRfc, minAmount, maxAmount.
• Paginacion: page (default 0), size (default 20).
• Ordenamiento: por createdAt descendente por default.

4.4 PATCH /api/v1/credit-applications/{id}/status - Cambiar estado
Request body:
{
 "status": "APPROVED",
 "reason": "Cumple politicas crediticias y tiene buen historial"
}
Transiciones validas:
• CREATED -> UNDER_REVIEW
• UNDER_REVIEW -> APPROVED
• UNDER_REVIEW -> REJECTED
• CREATED -> CANCELLED
• UNDER_REVIEW -> CANCELLED
Cualquier otra transicion: 409 Conflict con mensaje claro. APPROVED, REJECTED y CANCELLED son estados
terminales.
4.5 Formato de error estandar

 "code": "VALIDATION_ERROR",
 "message": "annualInterestRate must be between 0.05 and 0.60",

 updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
 CONSTRAINT chk_status CHECK (status IN

CREATE INDEX idx_credit_applications_status ON credit_applications(status);
CREATE INDEX idx_credit_applications_rfc ON credit_applications(customer_rfc);
CREATE INDEX idx_credit_applications_created_at ON credit_applications(created_at DESC);
Sobre el tipo de monto
Usa NUMERIC / BigDecimal para montos monetarios. Nunca double/float - introducen errores de
redondeo inaceptables en sistemas financieros.

6. Arquitectura hexagonal - estructura sugerida
La separacion en capas debe ser explicita en los paquetes. Sugerencia:
com.example.credit
 domain/
 model/ # CreditApplication, Money, CreditStatus
 exception/ # InvalidStatusTransitionException, etc.
 port/
 in/ # CreateCreditApplicationUseCase,
GetCreditApplicationUseCase, ChangeStatusUseCase
 out/
 CreditApplicationRepository.java # puerto de persistencia
 ExchangeRateProvider.java # puerto para Frankfurter
 application/
 service/ # implementacion de los use cases
 infrastructure/
 adapter/
 in/web/ # Controllers REST, DTOs, mappers, exception handler
 out/
 persistence/ # JPA entities, JpaRepository, Adapter
 external/ # FrankfurterExchangeRateAdapter
 config/ # ResilienceConfig, RestClientConfig, etc.
Regla clave
El paquete 'domain' NO debe importar nada de Spring, JPA, Jackson, ni de la libreria HTTP. Si lo
importa, no es hexagonal. La interfaz ExchangeRateProvider debe estar en domain.port.out. La
implementacion (que llama a Frankfurter) en infrastructure.

8. Pruebas unitarias requeridas
Como minimo deben existir pruebas para:
• Calculo de la cuota mensual (varios escenarios: tasas y plazos distintos, incluido tasa = 0 si
aplica).
• Logica de transicion de estados (no se puede pasar de CREATED a APPROVED, etc.).
• CreateCreditApplicationService: cuando el ExchangeRateProvider responde bien (mock), la
solicitud guarda los montos en USD/EUR. Cuando falla, la solicitud se crea con esos campos en
null.
• Validaciones de entrada (montos, plazos, RFC invalido).
• Manejo del status en el ChangeStatusUseCase (transiciones validas/invalidas).
Buenas practicas esperadas
Usar JUnit 5 + Mockito. Aplicar Arrange-Act-Assert. Probar casos felices y de error. Los tests del
dominio (entidad CreditApplication, calculo de cuota) NO deben depender de Spring. Solo Java puro
+ JUnit.

12. Banderas rojas (que NO hacer)
• Meter logica de negocio dentro del controller.
• Inyectar el JpaRepository directamente al service.
• Hacer la llamada HTTP a Frankfurter desde el service de aplicacion sin puerto.
• Usar H2 o base en memoria 'porque es mas rapido'.
• Usar ddl-auto=update en lugar de Flyway.
• Usar double/float para los montos (pierdes precision).
• Que un fallo de Frankfurter rompa la creacion de la solicitud.
• Subir credenciales reales o el target/ al repositorio.
• No tener pruebas o tener solo pruebas triviales (testear getters/setters).
• Devolver siempre 200 con un campo error - usar codigos HTTP correctos.
